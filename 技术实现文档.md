## 二、核心功能模块实现

### 2.1 AI 行程规划

**功能描述**：用户输入自然语言需求（如"我想去长沙玩3天，预算2000，喜欢历史文化"），系统调用通义千问AI生成完整的结构化行程。

#### 文件与方法链路

```
前端: client/src/components/AIChat.vue
  └─ sendMessage() → planAPI.generateItinerary(message)
       │
       ▼
后端: POST /api/plan
  文件: server/controllers/planningController.js
    方法: L152 ~ L200 (router.post('/plan', ...))

  执行流程:
    ① sessionManager.get(sessionId) → 获取/创建会话(TTL=2h)
    ② userService.getUserBySession(sessionId) → 获取userId
    ③ qwenAIService.generateTripFromNaturalLanguage(message, history, null, userId)
    ④ processAndSaveItinerary(aiResult, session, sessionId, null, userId)

  关键代码位置:
    planningController.js:L164 ~ L170  // userId获取(必须在AI之前)
    planningController.js:L172        // AI调用(传入userId加载偏好)
    planningController.js:L180 ~ L230  // processAndSaveItinerary 公共方法
```

#### AI 服务层

```
文件: server/services/ai/qwenAIService.js
  方法: generateTripFromNaturalLanguage(message, conversationHistory, weatherData, userId)

  实现流程:
    ① 构造 System Prompt:
       - 角色设定: "你是长沙旅游智能规划助手'小沙'"
       - 输出格式约束: 要求返回JSON格式(itinerary数组)
    
    ② 如果 userId 存在 → 加载用户偏好画像:
       preferenceLearningService.generatePreferenceProfile(userId)
       → 返回文本片段:
         "【用户偏好画像】偏好景点类型: 历史、自然; 偏好菜系: 湘菜..."
       → 注入到 System Prompt
    
    ③ 如果 weatherData 存在 → 注入天气上下文到 Prompt
    
    ④ 调用通义千问 DashScope API (POST /compatible-mode/v1/chat/completions)
       model: "qwen-plus"
       temperature: 0.7
    
    ⑤ 解析AI返回的JSON字符串，校验字段完整性
    
    ⑥ 返回: { ready, message, requirements, itinerary[], activities[] }
```

#### 统一后处理流水线

```
文件: server/controllers/planningController.js
  方法: processAndSaveItinerary(aiResult, session, sessionId, existingTripId, userId)

  该方法是 /plan 和 /chat 共享的后处理函数，消除重复代码:

  Step1: locationVerifyService.verifyItinerary(aiResult.itinerary)
    → 并行验证所有景点的真实经纬度(见2.2节)

  Step2: amapService.getWeather() 
    → 并行获取天气实况+预报(见2.3节)

  Step3: costCalculatorService.calculateTotalCost()
    → 分类计算费用并生成报告(见2.4节)

  Step4: smartPlanningService.enhanceItinerary()
    → TSP路线优化 + 交通推荐(见2.5节)

  Step5: tripService.createTrip() + saveTripRoutes()
    → 数据库持久化(见2.8节)

  返回值: { verifiedItinerary, costReport, enhancedPlanning, tripId }
```

### 2.2 POI 坐标验证

**功能描述**：AI生成的地点名可能是模糊或不准确的，需要通过高德地图API获取真实的经纬度和地址信息。

#### 文件与方法链路

```
文件: server/services/trip/locationVerifyService.js
  类: LocationVerificationService
  
  主方法: async verifyItinerary(itinerary)
    → 对每一天并行执行三类验证 (Promise.all):
        ├─ verifyAttractions(day.attractions[])
        ├─ verifyRestaurants(day.restaurants[])
        └─ verifyHotels(day.hotels[])

  子方法: async verifyAttractions(attractions)
    → 串行遍历每个地点(内部由amapService limiter控制并发)
    → amapService.searchPOI(attraction.name, '风景名胜')
    → 匹配结果:
        ├─ 精确匹配(name完全一致): 直接使用
        ├─ 模糊匹配(名称部分一致): 标记为 fuzzy_match
        └─ 无结果: 标记 unverified，保留原始名称

  子方法: verifyRestaurants() / verifyHotels()
    → 同上逻辑，搜索类型分别为 '餐饮服务' / '酒店宾馆'
```

#### 高德搜索服务

```
文件: server/services/external/amapService.js
  类: AmapService

  核心方法: searchPOI(keywords, city='长沙', type='')
    → GET https://restapi.amap.com/v3/place/text
    → 参数: key, keywords, city, type, output=JSON
    → 返回: [{ name, address, location(lng,lat), tel, ...}]

  并发控制:
    ConcurrencyLimiter 类 (内嵌在 amapService.js 中):
      maxConcurrent = 5     // 最大同时请求数
      maxQPS = 8           // 每秒最大请求次数
      minInterval = 120ms  // 最小请求间隔

  重试机制:
    失败时自动重试最多3次，指数退避:
      第1次重试: 等 800ms
      第2次重试: 等 1600ms
      第3次重试: 等 2400ms
    CUQPS(高德限流)错误码自动触发重试

  批量搜索统一方法:
    batchSearchPOIs(items, searchFn, city)
      → 将 N 个搜索任务加入 limiter 队列
      → Promise.all 并行等待全部完成
      → 返回按原顺序排列的结果数组

  便捷方法 (一行包装):
    searchAttractions(names) → batchSearchPOIs(names, searchPOI, '长沙', '风景名胜')
    searchRestaurants(names) → batchSearchPOIs(..., '餐饮服务')
    searchHotels(names)       → batchSearchPOIs(..., '酒店宾馆')
    searchSnacks(names)       → batchSearchPOIs(..., '美食')
    searchDrinks(names)       → batchSearchPOIs(..., '饮品')
    searchCheckInPoints(names) → batchSearchPOIs(..., '住宿')
```

### 2.3 天气数据集成

**功能描述**：获取长沙当前天气和未来4天预报，注入到每天行程中供参考。

#### 文件与方法链路

```
文件: server/services/external/amapService.js
  方法: async getWeather(city='长沙')

  实现流程:
    ① 并行请求两个高德天气接口 (Promise.all):
        
        接口A: GET /v3/weather/weatherInfo?key=&city=长沙&extensions=base
        → 返回实况: { temperature, winddirection, windpower, humidity, ... }

        接口B: GET /v3/weather/weatherInfo?key=&city=长沙&extensions=all
        → 返回预报: { forecasts: [
            { date, dayweather, nightweather, daytemp, nighttemp, ... },
            ...
          ]}

    ② 合并为统一格式:
        {
          current: { temp, weather, humidity, wind, ... },
          forecast: [
            { date, high, low, dayText, nightText, ... },  // 未来4天
            ...
          ]
        }
    
    ③ 在 processAndSaveItinerary 中将 forecast[i] 写入 itinerary[i].weather
```

### 2.4 智能费用计算

**功能描述**：对行程中所有项目的花费进行分类汇总、计算日均费用、生成预算分析报告。

#### 文件与方法链路

```
文件: server/services/trip/costCalculatorService.js
  类: CostCalculatorService

  主方法: calculateTotalCost(itinerary, requirements)
    → 返回: { total, breakdown, dailyCosts[], emergencyFund }

  计算规则:
    tickets (门票):
      Σ attraction.ticketPrice (免费景点 price=0)

    dining (餐饮):
      按天计算: 餐饮数量 × 人均消费 × 天数
      人均消费根据人群类型调整:
        独自旅行: 60元/餐
        情侣出游: 80元/餐  
        家庭亲子: 120元/餐
        朋友结伴: 70元/餐

    transport (交通):
      地铁: 4元/次 × 出行次数
      公交: 2元/次 × 出行次数
      步行/骑行: 0元
      网约车: 估算距离 × 2元/km

    hotel (住宿):
      hotel.pricePerNight 或 星级默认价:
        经济型: 200元/晚, 舒适型: 400元, 高档型: 800元, 豪华型: 1500元

    emergencyFund (应急金):
      total × 10% (向上取整到十位)

  辅助方法: generateCostReport(costData)
    → 返回饼图数据 + 各项占比文字说明
```

### 2.5 TSP 路线优化

**功能描述**：对AI生成的无序景点列表进行最优游览顺序排列，使总路程最短。

#### 文件与方法链路

```
文件: server/services/trip/smartPlanningService.js
  类: SmartPlanningService

  方法: optimizeRoute(attractions, startPoint=null)
    → 输入: N个景点对象(含latitude/longitude)
    → 输出: { route:[排序后数组], savedDistance(m), savedTime(min) }

  算法: 最近邻贪心法 (Nearest Neighbor Greedy)
    
    伪代码:
      起点 = startPoint 或 列表第一个元素
      已访问 = [起点]
      未访问 = [剩余N-1个]

      while 未访问不为空:
        当前点 = 已访问最后一个
        最近点 = 未访问中 距离(当前点, x) 最小的那个
        从未访问中移除最近点
        已访问.push(最近点)
      
      原始总距 = calculateTotalDistance(原始顺序)
      优化总距 = calculateTotalDistance(已访问)
      return { route:已访问, savedDistance:原始-优化, savedTime:(原始-优化)/80 }

  距离计算: calculateDistance(point1, point2)
    → Haversine球面余弦定理公式
    → R = 6371000m (地球半径)
    → 返回两点间地面直线距离(米)
```

### 2.6 交通方式推荐

**功能描述**：对行程中相邻的两个景点，根据多种因素评分推荐最佳交通方式。

#### 文件与方法链路

```
文件: server/services/trip/smartPlanningService.js
  方法: recommendTransportation(from, to, preferences={})

  支持的交通方式及评分因子:

    方式       | 适用距离    | 评分因子
    ----------|-----------|----------------------------------
    步行(walk)| < 1km     | 距离×1.0 + 天气影响 + 休闲加分
    骑行(bike)| 1~3km     | 距离×0.9 + 天气影响 + 健康加分
    地铁(subway)| > 3km    | 距离×0.7 + 可靠性 + 省时加分
    公交(bus)  | > 3km     | 距离×0.6 + 成本低 + 覆盖面广
    网约车(taxi)| 任意     | 舒适度×0.5 + 时间成本 - 价格惩罚

  评分公式(简化):
    score = base_score × distance_factor × weather_factor × time_preference_factor

    其中:
    - distance_factor: 超过舒适距离时分数递减
    - weather_factor: 雨/雪天步行和骑行大幅降分
    - time_preference: 根据偏好设置调整(快速/经济/舒适)
    - availability: 调用 _checkTransportAvailable() 检查地铁/公交实际可达性

  返回: [{type:'subway', score:92, duration:'15min', cost:4}, ...] 按score降序
```

#### 交通可用性检查

```
文件: server/services/external/amapService.js
  方法: checkSubway(lat, lon, radius=1000)
    → _searchNearbyStation('地铁', lat, lon, radius)
    → 返回 { available:true/false, stations:[], nearestDistance:m }

  方法: checkBus(lat, lon, radius=500)
    → _searchNearbyStation('公交', lat, lon, radius)
    → 同上逻辑

  通用搜索方法: _searchNearbyStation(type, lat, lon, radius)
    → 高德周边搜索 API (GET /v3/place/around)
    → 返回指定半径内的站点列表
```

### 2.8 数据持久化

**功能描述**：将生成的行程数据保存到MySQL数据库，支持后续查询、编辑、收藏等操作。

#### 文件与方法链路

```
文件: server/services/trip/tripService.js
  类: TripService

  创建行程: createTrip(userId, requirements, itinerary, history, routes, activities, sessionId)
    → INSERT INTO trips 表
    → 字段: id(UUID), user_id, session_id, title, requirements(JSON),
      activities(JSON), itinerary(LONGTEXT JSON), conversation_history,
      routes(LONGTEXT JSON), is_favorite, status, created_at
    → 返回: { tripId, success }

  保存路线: saveTripRoutes(tripId, itinerary)
    → 对每天每段相邻景点 INSERT INTO trip_routes

**功能描述**：预测指定景点在特定日期和时段的人流拥挤程度，帮助用户避开高峰。

**功能描述**：记录用户的浏览/收藏/点击等行为，通过加权模型构建用户偏好画像，用于个性化推荐和AI行程定制。

### 2.14 会话管理与内存安全

**功能描述**：管理用户会话状态，防止长时间运行导致内存泄漏。

#### 文件与方法链路

```
文件: server/controllers/planningController.js (内嵌定义)
  类: SessionManager

  设计动机:
    原方案: const sessions = new Map() → 无限增长 → OOM风险
    新方案: TTL过期 + 定时清理

  属性:
    sessions: Map<sessionId, { data, lastAccess }>
    SESSION_TTL_MS = 2小时
    CLEANUP_INTERVAL = 30分钟

  方法:
    get(id):
      读取 entry → 检查是否超TTL → 超期则delete并return undefined
      否则更新 lastAccess = Date.now() 并 return data

    set(id, data):
      写入 { data, lastAccess: Date.now() }

    _cleanup():
      每30分钟执行一次
      遍历所有session → 删除超过2h未访问的
      日志输出清理数量

  启动时创建唯一实例:
    const sessionManager = new SessionManager()
    cleanupTimer.unref()  // 不阻止进程退出
```

### 2.15 中间件与性能保障

#### 文件与方法链路

```
文件: server/middleware/index.js

=== 1. 请求频率限制 (Rate Limiter) ===

  配置:
    windowMs = 60000      // 时间窗口: 1分钟
    maxRequests = 600      // 窗口内最大请求数
    → 等效: 600/min = 10 QPS (远超5次/s要求)

  实现: 基于 IP + sessionId 的滑动窗口计数
  超限时返回: 429 Too Many Requests + Retry-After 头

=== 2. 异步日志 (Async Logger) ===

  文件: server/services/logger.js

  问题: 原方案 fs.appendFileSync 同步阻塞事件循环
  方案: WriteStream 异步写入 + 缓冲区批处理

  特性:
    - 按日期自动创建日志文件: logs/YYYY-MM-DD.log
    - 64KB缓冲区批量刷盘减少IO次数
    - 自动清理2天前的旧日志流
    - 提供 info/warn/error/debug 五个级别方法

=== 3. 全局错误处理 (Error Handler) ===
  捕获所有未处理的异常
  使用 logger.error() 记录
  返回统一格式: { code:500, message: error.message }

=== 4. 请求日志 (Request Logger) ===
  记录每个请求: method + url + status + 耗时(ms)
  使用 logger.info() 异步写入

## 五、性能优化措施总结

| 优化项 | 优化前 | 优化后 | 提升 |
|--------|--------|--------|------|
| POI批量搜索 | 串行+固定250ms delay | `ConcurrencyLimiter` 并发控制 | **4-5x提速** |
| 行程验证 | 串行(景点→餐厅→酒店) | `Promise.all` 三类并行 | **3x提速** |
| 交通推荐 | 串行循环 | `Promise.all` 全段并行 | **4x提速** |
| Logger | `fs.appendFileSync` 阻塞 | WriteStream 异步批处理 | **消除阻塞** |
| 会话存储 | `Map` 无限增长 | `SessionManager` TTL 2h+清理 | **防OOM** |
| 重试机制 | 无 | 3次+指数退避(800/1600/2400ms) | **容错提升** |
| 重复代码 | /plan 和 /chat 各写一遍 | 抽取 `processAndSaveItinerary` | **减50行冗余** |
| 6个搜索方法 | 各独立实现(~86行×6) | 统一 `batchSearchPOIs()` | **减80行冗余** |
